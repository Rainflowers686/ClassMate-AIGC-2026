## A. 产品定位（8 题）

### Q1：ClassMate 是不是一个课堂总结软件？
- 推荐回答：不是。ClassMate 是一个学习闭环工具：把课堂资料变成可验证的知识点、证据链、微测、自适应复习、练习入口和可导出学习报告。总结只是其中一步。
- 不该说：我们只是做 AI 总结。
- Proof：Timeline → Evidence → Quiz → Review → Practice → Report 完整录屏。

### Q2：你们和"听脑"类产品有什么区别？
- 推荐回答：听脑类产品核心是录音转写和摘要生成。ClassMate 的核心是学习闭环：知识点验证、证据溯源、微测、自适应复习、练习和可打印报告。输入只是起点，闭环才是终点。
- 不该说：竞品没有价值。录音转写类产品在记录和转写上很成熟。
- Proof：Review 任务页、Practice 搜索词卡片、可打印报告。

### Q3：为什么适合大学生？
- 推荐回答：大学生课后需要结构化复盘。课堂笔记可能零散，ClassMate 把课堂资料整理成带证据的知识点时间线，课后直接复习和自测。
- 不该说：所有学生都需要。
- Proof：课程库 + Timeline + Quiz 页面。

### Q4：为什么适合备考者（考研/考公）？
- 推荐回答：备考需要高频自测和错题追踪。ClassMate 的 Quiz 自动生成、Review 自适应复习、Practice 练习搜索入口，帮助备考者系统化查漏补缺。
- 不该说：可以替代真实真题训练。
- Proof：Quiz 微测页、Review 需要多练标签。

### Q5：为什么适合在职学习者？
- 推荐回答：在职学习时间碎片化。ClassMate 支持文本/图片/拍照快速导入培训材料，端侧模型可在通勤、弱网环境下提供本地智能问答和复习建议。
- 不该说：能替代系统培训。
- Proof：图片导入 → 端侧草稿 → Ask 页面。

### Q6：为什么叫"学习闭环"？
- 推荐回答：输入（课堂资料）→ 理解（知识点+证据）→ 自测（微测）→ 巩固（自适应复习）→ 拓展（练习搜索）→ 产出（可打印报告）。六个环节形成完整学习循环，而不是单次总结就结束。
- 不该说：闭环已经完美无缺。
- Proof：完整链路录屏。

### Q7：你们是否覆盖教师场景？
- 推荐回答：教师可以用 ClassMate 整理备课资料、生成课堂知识点提纲、导出可打印的学习资料分发给学生。但不是为课堂教学实时互动设计的。
- 不该说：能替代教师在课堂上的角色。
- Proof：Report 导出、Timeline 知识点整理。

### Q8：你们的目标用户画像是什么？
- 推荐回答：核心用户是 18–28 岁需要课后结构化复盘的学习者：大学生、考研/考公备考者、证书备考者和在职充电者。次要用户是教师和培训讲师。
- 不该说：覆盖所有教育场景。
- Proof：课程库用例多样性。

## B. 技术架构（9 题）

### Q9：云端蓝心和端侧蓝心如何分工？
- 推荐回答：云端 BlueLM 是主路径，负责课程分析、高质量问答、微测生成、报告生成等核心智能任务。端侧 BlueLM 3B 是本地智能兜底：当云端不可用时接管课程分析的本地执行，同时在 Ask/Report/Practice 中提供本地智能建议。
- 不该说：端侧能替代云端。
- Proof：Settings Provider 配置页 + provider path 日志。

### Q10：为什么端侧 BlueLM 是"本地智能兜底"而不是"本地替代"？
- 推荐回答：端侧 BlueLM 只有 3B 参数，回答质量受模型规模限制。它设计为兜底角色：云端正常时走云端，云端失败时端侧顶上，两种模型都失败时走安全占位，不生成假结果。
- 不该说：端侧和云端质量一样好。
- Proof：CourseAnalysis fallback path 截图。

### Q11：为什么 LocalRule 不再作为本地兜底？
- 推荐回答：在 Stage 8C 中，LocalRule 已降级为 Safety Placeholder（安全占位）。它的职责是：当云端蓝心和端侧蓝心都失败时，显示安全占位提示，明确告知用户"当前无法完成智能分析"，并建议检查网络和模型状态。不再生成任何看起来像 AI 的假结果。
- 不该说：LocalRule 还能做智能分析。
- Proof：安全占位 UI 截图。

### Q12：你们如何避免模型胡编（hallucination）？
- 推荐回答：三层防护：① 课程分析要求知识点绑定原文证据（Evidence quote + source marker）；② validators 校验 JSON 结构完整性和知识点/证据关联；③ 双模型失败时不生成假知识点，走安全占位。
- 不该说：模型绝对不会出错。
- Proof：validator failed 拦截截图 + 安全占位 UI。

### Q13：validators 具体做什么？
- 推荐回答：① JSON parse 校验：端侧输出必须是合法 JSON，否则不落库；② schema 校验：知识点必须包含 title/importance/difficulty/evidence 等必要字段；③ 证据校验：每个知识点必须有对应的原文引用；④ 范围校验：知识点来源必须匹配课堂资料。
- 不该说：validators 能纠正模型逻辑错误。
- Proof：validator 日志摘要。

### Q14：云端蓝心、端侧蓝心、安全占位三层的决策逻辑是什么？
- 推荐回答：① 优先走云端 BlueLM（质量最高）；② 云端失败（网络/超时/服务不可用）→ 自动切换到端侧 BlueLM 3B；③ 端侧也失败（模型未就绪/native crash/init 失败）→ 安全占位，不生成内容。每一步切换都有状态记录。
- 不该说：切换对用户完全透明无感知。
- Proof：provider path 展示、fallback 状态标签。

### Q15：端侧分析结果如何保证质量？
- 推荐回答：三层保证：① 同一套 validators 对端侧和云端结果同等校验；② 端侧 prompt 明确要求结构化 JSON 输出；③ 校验不通过的结果不落库，回退到安全占位。端侧通过的也标注来源为端侧蓝心，与云端结果区分。
- 不该说：端侧结果和云端一样可靠。
- Proof：端侧结果来源标签为端侧蓝心；双模型失败时显示安全占位。

### Q16：CourseAnalysis 主链如何过滤 LOCAL_FALLBACK？
- 推荐回答：CourseAnalysis 只在云端主路径失败后才尝试端侧 seam。端侧结果标记为端侧蓝心来源，在课程库和 Timeline 中与云端结果使用不同的来源标签展示；LOCAL_FALLBACK（旧 LocalRule）已被过滤，不再作为端侧结果。用户可以看到"本次分析由端侧模型完成"的提示。
- 不该说：用户看不到任何区别。
- Proof：课程详情来源标签。

### Q17：为什么选择 BlueLM 作为比赛模型？
- 推荐回答：这是 vivo 比赛规则要求的官方模型路径。BlueLM 是 vivo 自研大模型，我们使用官方提供的云端 API 和端侧 SDK 进行对接。这是比赛合规的基础。
- 不该说：BlueLM 是我们唯一支持的模型。
- 说明：架构上有 provider chain 预留扩展能力，但比赛展示走官方 BlueLM 通道。

## C. 端侧能力（10 题）

### Q18：如何证明端侧不是假接入？
- 推荐回答：四条证据：① 工程中有 OnDeviceBlueLM Provider 对接代码，通过反射桥接调用真实 SDK；② AAR 二进制存在且 gitignored（不入仓），可通过 javap 验证类签名；③ vivo X300 Pro 真机验证：Settings 诊断页显示 init/generate success；④ 多模态 callVit 返回 code 0、generate 成功。
- 不该说：代码里引用了 SDK 类名就是真接入。
- Proof：Settings 诊断截图、javap 输出、真机录屏。

### Q19：APK 为什么不内置模型？
- 推荐回答：端侧模型文件约 2–4 GB，内置到 APK 会导致安装包过大且不符合应用商店分发规范。模型文件存放在设备外部存储 `/sdcard/1225` 目录，由系统或用户预置。
- 不该说：模型下载是我们 App 的功能。
- Proof：APK 大小展示 + 模型目录截图。

### Q20：为什么模型路径是 `/sdcard/1225`？
- 推荐回答：这是 vivo 官方 BlueLM SDK 要求的模型存放路径。SDK 初始化时从此路径加载模型文件，路径不可由 App 自定义。
- 不该说：这是我们自己选的路径。
- Proof：SDK 文档引用（如有）+ 模型目录截图。

### Q21：为什么要 MANAGE_EXTERNAL_STORAGE 权限？
- 推荐回答：端侧 BlueLM SDK 需要从 `/sdcard/1225` 读取模型文件。在 Android 11+ 上，访问外部存储的特定目录需要 MANAGE_EXTERNAL_STORAGE 权限。这是功能必需的权限，不是过度索取。
- 不该说：我们没有申请这个权限。
- Proof：权限中心截图、Settings 诊断模型路径 OK。

### Q22：为什么 App 需要这么多权限？
- 推荐回答：每个权限都有真实功能对应：① CAMERA — 拍照导入课堂资料（板书/课件）；② RECORD_AUDIO — Live ASR 实验模式；③ MANAGE_EXTERNAL_STORAGE — 端侧 BlueLM 模型读取；④ POST_NOTIFICATIONS — 复习提醒；⑤ READ_MEDIA_* — 图片/文件导入。无一权限是未使用的。
- 不该说：都是系统要求加的。
- Proof：权限中心 + 功能对应说明。

### Q23：多模态 callVit 是什么？
- 推荐回答：callVit 是 BlueLM 端侧 SDK 提供的视觉编码接口。它将图片的 RGB 像素数据编码为模型可理解的特征向量，供端侧多模态推理使用。编码过程完全在设备本地执行，图片数据不离开设备。
- 不该说：callVit 就是 OCR。
- Proof：Settings 诊断 callVit code 0。

### Q24：端侧多模态和 OCR 有什么区别？
- 推荐回答：端侧多模态（callVit + 端侧推理）是对图片内容的整体理解——布局、结构、公式、图表关系——不逐字识别文字。OCR 是专门的文字提取技术。ClassMate 中多模态用于理解图片含义并生成可编辑草稿，不声称替代 OCR。用户在需要精确文字提取时可配合外部 OCR 工具。
- 不该说：多模态可以完整替代 OCR。
- Proof：端侧草稿编辑页 + 图片原文对照。

### Q25：端侧模型能在哪些设备上运行？
- 推荐回答：当前仅在 vivo X300 Pro（OriginOS 6 / Android 16 / arm64-v8a）上完成真机验证。端侧 BlueLM SDK 要求 arm64-v8a ABI、Android 8.0+（minSdk 28 实际最低要求）。不是任意手机都能跑。
- 不该说：任意 Android 手机都能跑。
- Proof：设备信息截图 + ABI 展示。

### Q26：端侧模型会不会让手机发烫或卡顿？
- 推荐回答：端侧推理会占用 CPU/NPU 资源，连续长时间使用可能导致设备发热。当前设计：① 端侧只做单次课程分析兜底，不做持续推理；② 简单问答走短文本推理，非长文本；③ 在 Settings 中有模型状态和诊断入口，用户可以了解模型运行状态。
- 不该说：端侧推理零资源消耗。
- Proof：Settings 模型诊断页。

### Q27：端侧模型是否支持所有课程类型的分析？
- 推荐回答：端侧 BlueLM 3B 受参数规模限制，对复杂课程（如高度专业的研究生课程、包含大量公式的数学推导）的分析质量不如云端。当前端侧主要用于：① 云端失败时的课程分析兜底；② 简单复习建议和练习方向建议。不是所有课程类型都适合端侧。
- 不该说：端侧适合所有课程。
- Proof：端侧分析结果来源标签和复杂度标注。

## D. 输入链路（8 题）

### Q28：文本、图片、拍照、录音、文件如何统一处理？
- 推荐回答：所有输入统一进入 MaterialBundle（资料篮），按来源标记（TEXT/IMAGE/CAMERA/TRANSCRIPT/FILE）。MaterialBundle 作为课程资料的统一中间层，之后进入 CourseAnalysis 生成知识时间线。
- 不该说：所有输入格式完美统一。
- Proof：Import Hub → Material Tray 截图。

### Q29：图片识别为什么要用户确认？
- 推荐回答：端侧多模态对图片的理解可能不准确。我们选择让用户先预览和编辑多模态生成的草稿内容，确认后再进入课程分析。这尊重用户判断，也避免不准确的内容自动污染知识库。
- 不该说：AI 识别结果不需要用户检查。
- Proof：草稿编辑页 + 确认按钮。

### Q30：拍照结果会不会自动进入知识库？
- 推荐回答：不会。拍照后流程是：① 端侧多模态生成可编辑草稿；② 用户预览并编辑草稿；③ 用户主动确认后进入课程分析；④ 课程分析生成的知识点经过 validators 才能入知识库。任何一步用户取消或不通过校验都不会入知识库。
- 不该说：拍照即自动分析。
- Proof：取消流程图 + History 记录对比 + validator 拦截。

### Q31：端侧多模态草稿可编辑是什么意思？
- 推荐回答：图片经过端侧 VIT 编码和推理后，生成对图片内容的文字描述草稿。用户可以在这个草稿上直接编辑、修改、补充内容。这相当于"AI 帮你看图写草稿，你自己改定稿"。
- 不该说：AI 生成的就是最终版本。
- Proof：草稿编辑页。

### Q32：图片/拍照入口在哪些页面？
- 推荐回答：Import Hub（导入中心）提供图片导入和拍照两个入口；课程详情页也支持追加图片资料。
- 不该说：所有页面都能拍照导入。
- Proof：Import Hub 截图。

### Q33：失败怎么处理？
- 推荐回答：分层处理：① 云端失败 → 尝试端侧 BlueLM 3B；② 端侧失败 → 安全占位（明确告知用户分析未完成）；③ 图片多模态失败 → 保留原始图片，提示用户手动输入文字描述或使用外部 OCR 工具。任何失败链路都不会生成假结果。
- 不该说：失败时会自动降级但结果一样好。
- Proof：failover 状态日志 + 安全占位 UI + 失败提示文案。

### Q34：你们支持哪些文件格式导入？
- 推荐回答：当前支持：纯文本粘贴、Markdown 文件、TXT 文本文件、SRT/VTT 字幕文件、图片（JPEG/PNG）、拍照。后续计划接入更多格式。
- 不该说：支持所有文件格式。
- Proof：Import Hub 格式列表。

### Q35：MaterialBundle 在架构中扮演什么角色？
- 推荐回答：MaterialBundle 是课程资料的统一中间抽象层。不同来源（文本、图片、字幕、Live 片段）进入 MaterialBundle 后统一打上来源标记，供下游的 CourseAnalysis、Ask、Quiz、Report 等模块使用。它解决了多源输入如何在统一链路中处理的问题。
- 不该说：这是一套完美的多模态融合方案。
- Proof：架构图 + 来源 marker 展示。

## E. 学习闭环（8 题）

### Q36：Timeline / Evidence / Ask / Quiz / Review / Practice / Report 如何串起来？
- 推荐回答：① 导入资料 → CourseAnalysis 生成 Knowledge Timeline（知识点时间线）；② 每个知识点绑定 Evidence（原文证据）；③ 基于 Timeline 可以使用 Ask（基于证据的问答）和 Quiz（微测）；④ Quiz 错题进入 Review（自适应复习）；⑤ Review 标记"需要多练"的知识点引导 Practice（练习搜索）；⑥ 最终所有内容可以导出为 Report（可打印学习报告）。
- 不该说：每一步都是 AI 完全自动的。
- Proof：完整链路录屏。

### Q37：Evidence（证据链）有什么价值？
- 推荐回答：① 学生可以追溯每个知识点来自课堂哪一段原文，不会学到"模型编的内容"；② 评委可以验证 ClassMate 不是黑箱输出——知识点必须有出处；③ 复习时不用重新看一遍整堂课，直接看关键原文。
- 不该说：证据链保证 100% 准确。
- Proof：Evidence quote + source marker 详情页。

### Q38：Quiz（微测）是怎么生成的？
- 推荐回答：基于 Knowledge Timeline 中的知识点，云端 BlueLM 自动生成选择题、判断题等微测题目。题目绑定对应知识点，做错的题目会进入 Review 模块。
- 不该说：微测题目质量超过专业出题老师。
- Proof：Quiz 题目页 + 绑定知识点展示。

### Q39：Review（自适应复习）的"自适应"体现在哪里？
- 推荐回答：① 基于 Quiz 答题结果标记掌握程度；② 答题错误的知识点进入"需要多练"；③ 复习频率根据知识点重要性和掌握程度调整；④ 不是简单重复所有内容，而是重点复习薄弱环节。
- 不该说：已经实现完整的间隔重复算法。
- Proof：Review 任务列表 + 需要多练标签。

### Q40：Practice（练习）和 Quiz（微测）有什么区别？
- 推荐回答：Quiz 是基于本课程知识点的封闭式微测（系统出题）。Practice 是开放式练习入口：系统生成搜索关键词，引导用户去外部教育资源平台（如题库、教学视频）进行针对性练习。Practice 不爬取第三方内容，只是搜索词建议。
- 不该说：Practice 内置了完整题库。
- Proof：Practice 搜索词卡片。

### Q41：Report 为什么叫"可打印学习资料"？

## F. 安全合规（9 题）

### Q44：AppKey/API Key 如何保护？
- 推荐回答：① 密钥只通过本地 debug 配置注入运行态（config.local.json，gitignored）；② 不入 Git 仓库、不入 CI 日志、不入截图、不入导出文件；③ secrets scan 脚本校验；④ 生产环境密钥部署有独立的安全通道。
- 不该说：密钥管理已完美解决。
- Proof：git check-ignore config.local.json + secrets scan 通过。

### Q45：AAR（端侧 SDK）为什么不入 Git 仓库？
- 推荐回答：① AAR 是 vivo 官方提供的二进制 SDK，属于第三方闭源组件；② 二进制文件体积大，不适合版本管理；③ 安全策略要求第三方二进制不进入源码仓库。构建时通过 local.properties 引用本地 AAR 路径。
- 不该说：AAR 已经提交到仓库。
- Proof：git check-ignore llm-sdk-release.aar + .gitignore 规则。

### Q46：端侧模型为什么不入 APK？
- 推荐回答：模型文件 2–4 GB，打包进 APK 会导致安装包过大，不符合 Google Play 150MB 限制和 vivo 应用商店分发规范。模型文件存放在设备 `/sdcard/1225` 目录，独立于 APK。
- 不该说：模型已经在 APK 里。
- Proof：APK size 展示 + 模型目录截图。

### Q47：图片是否上传到云端？
- 推荐回答：不。图片处理全链路在设备本地：① 用户选择/拍摄图片 → ② 端侧 VIT 编码（本地） → ③ 特征向量供端侧模型推理（本地） → ④ 推理结束后原始图片数据释放。图片和特征向量均不经过网络传输。
- 不该说：图片会在云端做 OCR。
- Proof：断网场景多模态仍可运行 + 网络抓包可验证。

### Q48：是否保存原始音频？
- 推荐回答：不。Live ASR 实验模式使用系统 SpeechRecognizer 进行实时语音识别，识别结果以文本形式保存，原始音频数据不落盘持久化。
- 不该说：音频已完全丢弃但 ASR 转写完美。
- Proof：Live 页面隐私说明 + 存储权限审计。

### Q49：是否爬取第三方教育平台内容？
- 推荐回答：不。Practice 模块只生成搜索关键词建议，由用户自行选择是否去外部平台搜索。ClassMate 不做任何形式的网页爬取、内容抓取或平台数据采集。
- 不该说：我们有题库聚合能力。
- Proof：Practice 搜索词卡片 + 用户手动搜索流程。

### Q50：是否泄露 prompt 或模型的 reasoning_content？
- 推荐回答：不。① 模型交互日志只记录短字段（provider/status/latency），不记录 prompt 全文和模型原始响应；② reasoning_content（如有）不会被保存、展示或导出；③ Export 有 redaction 机制，自动移除敏感类别。
- 不该说：完整日志供调试使用。
- Proof：日志短字段展示 + Export 安全检查截图。

### Q51：你们如何处理用户数据隐私？
- 推荐回答：① 所有学习数据存储在设备本地，不强制云同步；② 端侧模型推理数据全链路不出设备；③ 云端分析只传输必要的课程文本（非用户身份信息）；④ 导出前有 redaction 检查。
- 不该说：用户数据完全不需要网络。
- 说明：云端分析需要传输课程文本，但这是功能必需的，且不含用户个人信息。

### Q52：安全占位（Safety Placeholder）和错误提示有什么区别？
- 推荐回答：安全占位是"诚实声明"：明确告知用户当前模型不可用，建议检查网络和模型状态，不伪装成 AI 成功。错误提示是技术层面的异常信息（如"网络连接超时"）。安全占位是产品层的用户可见状态，比技术错误更具可读性和操作指导性。
- 不该说：安全占位就是错误提示。
- Proof：安全占位 UI 截图。
