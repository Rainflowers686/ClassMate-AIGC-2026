# Stage 8A-4 端侧真机失败答辩话术

> 端侧模型在真机上可能出现各种失败场景。此文档为答辩时应对评委提问的口径。
> 策略：诚实说明技术原因，将"失败"转化为"证明了 fallback 机制设计的正确性"。
> 不声称端侧真机已跑通。

---

## 1. SDK 初始化失败

- **触发场景**：`LlmManager.init()` 返回非 0
- **评委可能问**："你的端侧模型不是说接好了吗，为什么初始化失败？"

**话术**：
> "端侧模型的初始化依赖多个条件：SDK 文件存在、native library 加载成功、模型文件在设备上可用、设备 SoC 支持 NPU/APU。
> 当前设备上 init 返回了错误码 {code}，这正好证明了我们的——系统**真实地**在调用 SDK API，而不是返回一个 hardcoded 的'成功'。
> 同时也证明了我们的 fallback 机制：端侧初始化失败后，系统自动切换到云端 BlueLM，用户功能完全不受影响。
> 端侧模型的设备兼容性是已知限制——BlueLM 3B SDK 有设备白名单，部分机型不在支持范围内。这是厂商 SDK 层面的约束，不是我们代码的问题。"

---

## 2. 模型路径不存在

- **触发场景**：`/sdcard/1225/` 下没有模型文件
- **评委可能问**："你的模型文件在哪里？"

**话术**：
> "端侧模型文件需要提前下载到设备上。当前设备还没有放置模型文件——这不是技术问题，是部署流程的问题。
> 在生产环境中，端侧模型文件通过应用内下载分发，存储在应用私有目录。
> 由于比赛使用的云真机是临时分配的，模型文件没有提前预置。但我们的代码中对路径不存在的情况有明确的错误处理，显示为 `MODEL_PATH_NOT_FOUND`，而不是崩溃或假输出。
> 这也是为什么我们设计了 fallback——模型文件缺失时，系统自动使用云端模型。"

---

## 3. 权限不足

- **触发场景**：应用没有存储读取权限
- **评委可能问**："为什么端侧不能用？是代码问题吗？"

**话术**：
> "端侧模型需要读取模型文件，因此需要存储权限。当前设备的权限状态限制了模型文件的访问。
> 这恰好证明了我们的权限检查是**真实的**——应用会验证权限是否满足，而不是假设权限永远可用。
> 我们的 Manifest 权限策略也经过了审计——端侧功能只申请了存储权限和网络权限，没有额外申请位置、通讯录等权限。"

---

## 4. Native Library 加载失败

- **触发场景**：`System.loadLibrary` 失败
- **评委可能问**："你的 SDK 是不是有问题？"

**话术**：
> "Native library 加载失败通常与设备 ABI 不匹配有关。BlueLM SDK 的 .so 文件支持 arm64-v8a 架构，如果云真机是 x86 模拟器就无法加载。
> 这是 Android NDK 开发的常见限制，不是 SDK 缺陷。我们在 Settings 诊断页面中会明确显示 `NATIVE_LIBRARY_LOAD_FAILED` 状态和错误信息。
> 在真实的 arm64 设备上（如 vivo X300 Pro），native library 加载是正常的。云真机的架构限制不在我们控制范围内。"

---

## 5. NPU/APU 不可用

- **触发场景**：设备 AI 加速器不可用或未配置
- **评委可能问**："端侧模型的硬件加速呢？"

**话术**：
> "端侧模型利用设备的 NPU（神经网络处理器）或 APU（AI 处理单元）来加速推理。不是所有设备都具备这些硬件加速器。
> 当 NPU 不可用时，SDK 可能有 CPU fallback，但推理速度会显著降低。我们在 `LlmConfig` 中配置了 `npuPower` 参数来管理这个行为。
> 这是端侧推理的硬件现实——我们诚实地反映了设备能力，而不是伪造一个'加速成功'的状态。"

---

## 6. callVit 失败

- **触发场景**：`LlmManager.callVit()` 返回非 0
- **评委可能问**："多模态功能不能用？你们展示了什么？"

**话术**：
> "端侧多模态是我们的**实验性功能**。callVit（Vision Transformer 编码器）的可用性取决于设备是否支持端侧 VIT 推理。
> 在之前的演示中，我们已经明确说明多模态是实验性质、尚未进入学习主链路。callVit 失败的情况我们在 error copy 文档中有完整覆盖（20 条错误文案，编号 #15/#16/#17 针对多模态）。
> 多模态的主力方案仍然是云端 BlueLM，端侧 VIT 是**隐私优先场景下的补充**——例如学生拍了一道题但不希望题目图片上传到云端的情况。"

---

## 7. 多模态 generate 失败

- **触发场景**：callVit 成功但 generate 未输出
- **评委可能问**："图片编码成功了但为什么没有结果？"

**话术**：
> "端侧多模态的 generate 阶段需要特殊的多模态 prompt 模板——需要在 prompt 中包含 image token 占位符来告诉模型'这里有图片信息要处理'。
> 如果 prompt 模板不正确或模型不支持多模态推理模式，generate 可能不输出或输出不相关的内容。
> 多模态 prompt 工程是一个独立的研究方向，我们目前完成了 VIT 编码的桥接，但 prompt 模板的优化是需要进一步迭代的工作。"

---

## 8. 断网下端侧不可用

- **触发场景**：断网后端侧模型也不可用（双重不可用）
- **评委可能问**："你说端侧可以离线用，现在呢？"

**话术**：
> "端侧模型的离线可用性有两个前提：第一，SDK 初始化成功；第二，模型文件已加载到内存。如果初始化本身就失败了（比如因为前面提到 native library 或模型路径问题），那断网后当然也不可用。
> 但请注意，我们的系统在这种情况下仍然**不会白屏或崩溃**——LocalRule 本地规则引擎作为最后的兜底层会接管请求，处理简单的预设问答。
> 我们设计的是**三层兜底**：端侧 → 云端 → 本地规则。断网砍掉了中间层，最底层仍然在。"

---

## 9. fallback LocalRule 生效

- **触发场景**：端侧和云端都不可用，LocalRule 启用
- **评委可能问**："你们最后就是用一个规则引擎在应付？"

**话术**：
> "LocalRule 是我们的**最后的兜底层**，不是我们的主要功能。它的存在是为了保证在最极端的情况下——比如设备在飞行模式且端侧模型初始化失败——应用仍然能给用户一个基本的、有用的响应。
> 正常情况下，云端 BlueLM 和端侧 BlueLM 3B 会覆盖绝大多数场景。LocalRule 只在双重不可用时激活。
> 这实际上展示了我们系统设计的**鲁棒性**——每一层都有预案，不会因为单个组件的失败导致整个系统不可用。"

---

## 10. 如何诚实说明"已完成 bridge，真机环境待适配"

**评委可能问**："你到底做完了没有？"（综合性质疑）

**话术**：
> "让我诚实地说明我们完成了什么、还有什么在规划中：
>
> **已完成**：
> 1. 真实 SDK bridge 的代码实现——通过 reflection 桥接，支持 LlmManager 的所有核心方法
> 2. 端侧 Provider 的 ProviderChain 集成
> 3. Settings 诊断页面——实时显示 SDK 状态、初始化结果、错误码
> 4. TokenCallback 流式输出的正确对接（使用新的无参 onComplete 签名）
> 5. 多模态 callVit 的实验性桥接
> 6. fallback 机制的完整链路
> 7. 全套文档、测试计划、评委问答准备
>
> **需要真机环境适配的**：
> 1. 端侧模型文件的预置或下载
> 2. 特定设备 SoC 的兼容性验证
> 3. 端侧推理性能的基准测试
> 4. 多模态 prompt 模板的调优
>
> 这些'待适配'项**不是代码缺陷**——我们的代码对接是正确的，javap 验证了 API 签名匹配，Settings 诊断页面证明了 SDK 状态检测是真实的。真机适配是部署流程的一部分，需要设备厂商的环境配合。"

---

## 通用原则

1. **不掩饰失败**：失败就是失败，诚实说"当前设备端侧不可用"
2. **转化为论据**：每次失败都证明 fallback 设计正确
3. **引用证据**：javap 签名、Settings 诊断状态、error code 都是有说服力的技术证据
4. **预先管理期望**：在所有演示文档中提前写"如果端侧失败，请看 backup script"
5. **不伪造成功**：永远不 hardcode "成功"状态
