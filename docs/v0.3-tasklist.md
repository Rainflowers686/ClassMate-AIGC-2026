# v0.3 — entering main-flow skeleton

Goal: turn the v0.2.5 probe into a runnable end-to-end demo of the 12 must-do
features (spec §2.1 R1–R12), still on Android, still single-provider-at-a-time,
still demo-fallback safe.

## Acceptance gate (do not declare v0.3 done until all pass)

- A1 main flow: tap from Home → Course Input → Hotword → Analyze → Timeline → Quiz → Review Plan, no crashes.
- A2 model call: at least one real call to BlueLM **or** a compatible provider succeeds end-to-end (cloud-emulator screenshot in `proof/`).
- A3 JSON: response validates against `schema/course_analysis_result.schema.json`.
- A4 render: UI is driven by parsed result, no static placeholder content.
- A5 evidence chain: tapping a knowledge point or quiz reveals the source segment and `evidence_span`.
- A6 quiz: answering affects knowledge-point status (correct/incorrect tint).
- A7 review plan: generated from quiz state + difficulty/importance, not hardcoded.
- A8 fallback: simulated network-off / 5xx falls back to demo data, no crash.
- A9 logs: every model call produces one English redacted record.
- A10 README: an unfamiliar reader can reproduce the demo flow.
- A11 keys: no real key in git; `git log -p` clean.
- A12 video: cloud emulator screen-capture can be recorded.

## Task list (ordered)

### 1. Provider HTTP layer
- [ ] Add OkHttp / Ktor client to `core` (keep core JVM-only — add a small `Network` abstraction so app can inject the engine).
- [ ] Implement `BlueLMProvider.analyzeCourse` against the documented BlueLM endpoint.
  - Sign with `app_id` / `app_key` per BlueLM docs.
  - Stream-or-buffer is a v0.3 decision; default to buffered until quizzes start arriving slow.
- [ ] Implement `CompatibleProvider.analyzeCourse` against an OpenAI chat-completions shape.
- [ ] Both providers MUST emit one record via `RedactedLogger` per call.
- [ ] Both providers MUST throw a typed exception (`ModelCallException`) on failure so the UI can fall back to demo without `catch (Throwable)`.

### 2. Schema validator
- [ ] Add a `JsonSchemaValidator` to `core/schema/` (consider `com.networknt:json-schema-validator`; if Android-incompatible, hand-roll for the fields we actually require).
- [ ] Call it from every provider before returning.
- [ ] Validation failure must hit the demo-fallback path and log `schema_valid=false`.

### 3. Segmenter
- [ ] Implement the priority rules from spec §9: timestamp split → paragraph merge to 150–300 chars → punctuation+length fallback.
- [ ] Unit-test on three fixtures: timestamped lecture transcript, blog-style paragraphs, single run-on string.

### 4. UI screens (Compose, one ViewModel per screen)
- [ ] `CourseInputPage` — paste text or pick `demo_course.txt`.
- [ ] `HotwordPage` — chip input, 5–20 items, validates count.
- [ ] `AnalyzePage` — progress + cancel.
- [ ] `TimelinePage` — knowledge-point cards grouped by segment.
- [ ] `QuizPage` — one quiz at a time, scored, marks related KP.
- [ ] `ReviewPlanPage` — generated from quiz state.
- [ ] Single `NavHost`; back stack survives rotation.

### 5. Evidence-link UI
- [ ] Tapping a knowledge-point card opens a bottom sheet showing `evidence_span` highlighted in the source segment.
- [ ] If `evidence_span` mismatched: show segment context, no highlight, footnote "model-quoted snippet not found verbatim".

### 6. State + persistence
- [ ] `CourseSessionRepository` — in-memory only is fine for v0.3; persist if cheap.
- [ ] `QuizStateManager` — tracks answered/correctness per quiz_id.
- [ ] `ReviewPlanner` — composes plan from quizState + difficulty + importance + confusion_points.

### 7. Demo fallback
- [ ] `DemoFallbackManager` decides when to swap providers (network fail, schema fail, evidence chain fully broken).
- [ ] Show a non-blocking banner: "当前模型服务暂不可用，已加载本地演示数据".

### 8. Proof artifacts
- [ ] One real `api_call_*_redacted.jsonl` checked into `proof/logs_redacted/`.
- [ ] One cloud-emulator screenshot per main-flow screen in `proof/screenshots/`.
- [ ] One short cloud-emulator screen recording.

### 9. Hygiene
- [ ] CI step that fails the build if any committed file contains a string matching the App Key regex (length + charset heuristic).
- [ ] CI step that runs the schema validator over `examples/demo_output.json`.
- [ ] Gradle task `copyAssets` that syncs `schema/`, `prompts/`, `examples/` into `app/src/main/assets/`, so the two trees can no longer drift.

## Out of scope for v0.3

Same as v0.2 spec §3.1 — no login, no community, no real Small V / atom-notification / negative-one-screen integration, no on-device LLM, no wearable, no long-term study profile, no multi-step async pipeline.
